# Fundamentals - Questions and Answers

1. **What is open source software?**

   Open source software (OSS) is software with source code that anyone can inspect, modify, and enhance. It's built on the principles of collaboration, transparency, and community-oriented development.

---

2. **Explain the difference between popular licenses: MIT, Apache 2.0, and GPL.**
   - **MIT**: A very permissive license. You can do almost anything as long as you provide attribution.
   - **Apache 2.0**: Similar to MIT but includes an explicit grant of patent rights.
   - **GPL (General Public License)**: A copyleft license. If you use GPL code in your project and distribute it, your project must also be open-source under the same GPL license.

---

3. **What is a "Fork" and how does it differ from a "Branch"?**
   - **Fork**: A personal copy of someone else's project that lives in your account. It's the starting point for contributing to a project you don't have write access to.
   - **Branch**: A temporary path of development within a single repository. Used to work on specific features or fixes within the same codebase.

---

4. **How do you submit a Pull Request (PR)?**
   1. Fork the repository.
   2. Clone your fork locally.
   3. Create a new branch for your changes.
   4. Commit and push your changes to your fork.
   5. Open a Pull Request from your branch to the original repository's main branch.

---

5. **What are common best practices for contributing to an open source project?**
   - Read the `CONTRIBUTING.md` file first.
   - Open an issue before making major changes.
   - Keep PRs small and focused on one thing.
   - Use clear, descriptive commit messages.
   - Be respectful and follow the project's Code of Conduct.

---

6. **Explain the importance of a `README.md` and `CONTRIBUTING.md`.**
   - **`README.md`**: Provides an overview of the project, how to install it, and how to use it. It's the "front door" of the project.
   - **`CONTRIBUTING.md`**: Outlines the process for submitting bugs, feature requests, and code. It helps maintainers set expectations for contributors.
